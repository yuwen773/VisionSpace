# 行为聚合系统改进设计

## 1. 背景

当前行为聚合存在以下问题：
- 全量扫描，性能随数据量增长而恶化
- action 数据只增不清，数据量不断膨胀
- PictureStats 实体存在但未使用
- picture 表承担了统计职能，职责不清晰

## 2. 目标

- 增量聚合，避免全表扫描
- action 数据保留 7 天，定期清理
- 启用 PictureStats 作为主统计表
- picture 表精简为纯基本信息表

## 3. 数据模型

### 3.1 picture 表（精简）

移除统计字段：
- likeCount, viewCount, collectCount, downloadCount, shareCount, impressionCount, ctr

保留字段：
- id, url, thumbnailUrl, previewUrl, name, introduction, category, tags
- picSize, picWidth, picHeight, picScale, picFormat, picColor
- userId, spaceId, reviewStatus, reviewMessage, reviewerId, reviewTime
- createTime, editTime, updateTime, isDelete

### 3.2 picture_stats 表（主统计表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| pictureId | BIGINT | 图片ID |
| viewCount | BIGINT | 查看次数 |
| likeCount | BIGINT | 点赞次数 |
| collectCount | BIGINT | 收藏次数 |
| downloadCount | BIGINT | 下载次数 |
| shareCount | BIGINT | 分享次数 |
| impressionCount | BIGINT | 曝光次数 |
| ctr | DECIMAL(10,6) | 点击率 = clickCount / impressionCount |
| createTime | DATETIME | 创建时间 |
| updateTime | DATETIME | 更新时间 |

### 3.3 user_picture_action 表（行为明细）

新增字段：
- isProcessed TINYINT DEFAULT 0 — 标记是否已聚合

索引优化：
- 已有 idx_picture_time(pictureId, createdTime DESC)
- 新增 idx_isProcessed(isProcessed)
- 新增 idx_createdTime(createdTime)

## 4. 聚合流程

### 4.1 定时任务（每 5 分钟）

```
1. 查找近5分钟内的 action 记录
   SELECT * FROM user_picture_action
   WHERE createdTime > #{lastRunTime} AND isProcessed = 0

2. 按 pictureId + actionType 聚合
   GROUP BY pictureId, actionType
   SUM(actionValue)

3. UPSERT 到 picture_stats
   - 如果 pictureId 存在则 UPDATE
   - 如果不存在则 INSERT

4. 标记已处理
   UPDATE user_picture_action SET isProcessed = 1
   WHERE createdTime <= #{currentRunTime} AND isProcessed = 0

5. 清理7天前数据（可选，异步执行）
   DELETE FROM user_picture_action
   WHERE createdTime < DATE_SUB(NOW(), INTERVAL 7 DAY)
```

### 4.2 配置

```yaml
picture-stats:
  aggregate:
    interval-ms: 300000  # 5分钟
    batch-size: 500
  retention:
    days: 7
```

## 5. 查询变化

### 5.1 图片详情查询

之前：
```sql
SELECT *, likeCount, viewCount FROM picture WHERE id = ?
```

现在：
```sql
-- 需要 JOIN picture_stats
SELECT p.*, ps.likeCount, ps.viewCount, ps.ctr
FROM picture p
LEFT JOIN picture_stats ps ON p.id = ps.pictureId
WHERE p.id = ?
```

### 5.2 图片列表查询

需要聚合查询或视图支持。

## 6. 实施步骤

### Phase 1: 数据结构变更
1. 新增 picture_stats 表
2. 新增 user_picture_action.isProcessed 字段和索引
3. 修改 picture 表，移除统计字段

### Phase 2: 服务层改造
1. 修改 PictureStatsAggregateService 实现增量聚合
2. 修改 PictureService 相关查询，支持 JOIN picture_stats
3. 修改定时清理任务

### Phase 3: 数据迁移
1. 将 picture 表现有统计数据迁移到 picture_stats
2. 验证数据一致性

### Phase 4: 清理
1. 删除 picture 表统计字段（确认无使用后）
2. 删除旧的聚合代码

## 7. 风险与注意事项

- 查询性能：列表页需要 JOIN，可能需要缓存或索引优化
- 数据一致性：迁移过程需注意双写或断服迁移
- 兼容性：API 返回格式变化需通知前端
